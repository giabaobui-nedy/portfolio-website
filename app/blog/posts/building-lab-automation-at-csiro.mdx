When I joined CSIRO as an intern, the research team was running experiments the old-fashioned way — manually operating hardware, recording readings by hand, and hoping nothing went wrong overnight. By the time I left, they had a fully automated system that could run experiments safely, track everything in real time, and let scientists monitor progress from their desks.

Here's what I learned building it.

## The Problem

The lab operated chromatography columns that required precise control of pumps, valves, and sensors. Researchers had to be physically present to adjust parameters, and there was no centralised record of what happened during each experiment. This meant:

- **No traceability** — if an experiment failed, it was hard to figure out why
- **No remote monitoring** — scientists had to be on-site at all times
- **No safety automation** — if a pressure reading spiked, someone had to notice manually

## The Architecture

I designed the system as a layered stack: **UI → API Server → Firmware Abstraction → Modbus Hardware**.

The key insight was building a **firmware abstraction layer** that decoupled the software from specific hardware configurations. Instead of hardcoding each pump and valve, I created configuration-driven patterns that could adapt as the lab's multi-column setups evolved.

```python
class HardwareController:
    """Base controller with common Modbus communication logic."""

    def read_register(self, address: int) -> int:
        with self._lock:
            return self._client.read_holding_registers(address).registers[0]
```

Threading was critical — multiple users might try to control hardware simultaneously. I implemented lock mechanisms to prevent concurrent access and firmware-level caching to minimise unnecessary hardware calls. This contributed to three months of uninterrupted production operation.

## Real-time Telemetry

One of the most rewarding parts was the monitoring dashboard. I integrated **InfluxDB** for time-series telemetry data and built dynamic **ECharts** visualisations with WebSocket updates. Scientists could see live pressure, flow rate, and UV readings, filter by time range, and overlay multiple experiments for comparison — all without refreshing the page.

## Security and Access Control

Working within CSIRO's institutional environment meant security wasn't optional. I implemented **Azure Entra ID SSO** with middleware-based role authorisation, ensuring that only authorised researchers could control hardware while others could view experiment data in read-only mode.

## What I Took Away

This project taught me that building software for scientists is fundamentally about trust. They need to trust that the system won't damage expensive equipment, that the data it records is accurate, and that it will behave predictably under every condition.

That means investing heavily in:

- **Defensive error handling** at every layer
- **Structured logging** so problems can be diagnosed after the fact
- **Clear API boundaries** that make the system debuggable
- **Thorough documentation** so the next engineer can maintain it

The system is still running in production today, used by 15–20 active scientists. It's the project I'm most proud of — not because the code is perfect, but because it genuinely changed how research gets done.
