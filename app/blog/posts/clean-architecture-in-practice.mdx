Clean Architecture is one of those concepts that sounds great in theory but gets messy when you actually try to implement it. When I built a Weekly Planner application in Java, I decided to go all-in on the Ports & Adapters pattern to see if the investment was worth it.

It was.

## The Core Idea

The central rule of Clean Architecture is that **domain logic should not depend on infrastructure**. Your business rules shouldn't know or care whether data comes from SQLite, PostgreSQL, or an in-memory store. They shouldn't know whether they're being called from a GUI, a CLI, or a REST API.

In practice, this means:

- **Domain entities** live in the centre and have zero external dependencies
- **Use cases** orchestrate domain logic through interfaces (ports)
- **Adapters** implement those interfaces with concrete technology (database, UI, etc.)

## How It Worked in the Planner

The planner's core use case was a **rule-based scheduling algorithm** that assigned tasks to weekly time slots based on priority, constraints, and user-defined working hours.

The domain layer defined what a `Task`, `Schedule`, and `TimeSlot` looked like — pure data and logic, no imports from JavaFX or SQLite:

```java
public interface TaskRepository {
    List<Task> findByWeek(LocalDate weekStart);
    void save(Task task);
    void delete(UUID id);
}
```

The infrastructure layer provided the actual implementation:

```java
public class SqliteTaskRepository implements TaskRepository {
    private final Dao<TaskEntity, UUID> dao;

    @Override
    public List<Task> findByWeek(LocalDate weekStart) {
        return dao.queryForEq("weekStart", weekStart)
            .stream()
            .map(TaskMapper::toDomain)
            .collect(Collectors.toList());
    }
}
```

## Why It Paid Off

**Testing became trivial.** I could test the scheduling algorithm with a simple in-memory implementation of `TaskRepository` — no database setup, no teardown, no flaky tests.

**Swapping infrastructure was real.** When I needed to change how tasks were persisted, I only touched the adapter layer. The domain logic and use cases didn't change at all.

**The codebase scaled cleanly.** Adding new features meant adding new use cases and adapters, not modifying existing ones. Each layer had a clear responsibility.

## The Trade-off

The honest truth: Clean Architecture adds boilerplate. For a small app, you might wonder if the extra interfaces and mappers are worth it. But the moment your project grows past a handful of features — or the moment you need to write tests — you'll be grateful for the separation.

The Weekly Planner taught me that architecture isn't about following rules for their own sake. It's about making future changes cheap and safe. That's a lesson I carry into every project now.
